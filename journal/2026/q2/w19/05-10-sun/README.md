## Agent index

- Z3WIRE-COMPARISON-STRICT-OPERATORS (s01): `==`/`!=`/`<`/`<=`/`>`/`>=` on `SymBitVec<W,S>` now require matching width AND signedness via two-parameter template `<W1,S1,W2,S2>` + `static_assert(W1==W2 && S1==S2, "comparison requires matching width and signedness; use z3w::math_eq(a, b) for mathematical equality, or cast one operand to the other's type")`. Two-param template form keeps the diagnostic targeted (not generic "no matching overload"). Concrete `BitVec` and `Bool`/`SymBool` untouched
- Z3WIRE-MATH-COMPARISON-FAMILY (s01): heterogeneous mathematical (value-based) comparison as free fns `z3w::math_eq`/`math_ne`/`math_lt`/`math_le`/`math_gt`/`math_ge`. Sym×sym + mixed concrete+sym overloads on `math_eq`/`math_ne`. Derived three (`math_le`/`math_gt`/`math_ge`) delegate to `math_lt`. `math_ne` refactored to `!math_eq` post-tasks for consistency. C++20 `std::cmp_*` is direct precedent
- Z3WIRE-COMPARISON-DESIGN-RATIONALE (s01): old behavior mathematically correct but silently traversed type boundaries, conflicting with library's "explicit over implicit" principle (bitwise already strict, arithmetic uses bit-growth). For `==`, `SymUInt<8>(255) == SymSInt<8>(-1)` is `false` mathematically vs `true` bit-pattern — strict matching forces explicit choice. Cross-lang precedent: Rust/Go/Haskell/Zig all strict; Swift went heterogeneous-for-all-six and has documented overload-res + integer literal overflow problems
- Z3WIRE-PACKET-VIEW-DESIGN (s02): 4 class templates each templated on `(int BufW, int BaseOffset)`, holding `z3w::SymUInt<BufW>* buf_`. `Ipv4AddrView` (full 32 bits + `octet0..3` accessors), `Ipv4View` (leaf fields + `src_addr`/`dst_addr` sub-views at local offsets +32/+0), `EthView` (leaf fields), `HdrView<BufW>` composite (no own `buf_`, holds `EthView<BufW, 320>` + `Ipv4View<BufW, 160>`). Local bit indices are LSB-relative within each view; `extract<Hi + BaseOffset, Lo + BaseOffset>` + `replace<Lo + BaseOffset>`. Closest cousins: Chisel `Bundle`, SystemVerilog packed structs, P4 headers, CMSIS register access
- Z3WIRE-NODISCARD-CONVENTION (s02): state-returning accessors get `[[nodiscard]]` (e.g. `z3w::SymBitVec::expr()`). Enforced by clang-tidy `modernize-use-nodiscard` as warning-as-error. Phase 3 lint surfaced 12 misses → bake into plan from Phase 1 next time
- CONSTRUCTOR-INIT-ORDER-CPP (s02): C++ language rule — init order must match member declaration order regardless of init-list order. `-Wreorder-ctor` warning only surfaces on template instantiation. In `Ipv4View`, fix was `: src_addr(buf), dst_addr(buf), buf_(buf) {}` matching declaration order. Bazel cache hit was so good after fix (action hash unchanged because cosmetic) that build output didn't reveal whether fix took effect — trust language semantics, verify with spec reviewer
